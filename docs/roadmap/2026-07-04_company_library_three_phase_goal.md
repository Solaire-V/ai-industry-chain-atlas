# 公司库三阶段目标

## 最终目标

把 AI 全产业链图谱升级为可公开访问的网站。公司库是面向产业链研究者和投资者的核心查询入口：用户可以按产业链节点、市场、公司角色、行情估值和供需关系快速定位公司，并从公司反查其所在产业链位置。

桌面端是第一优先级。手机端不作为当前核心体验，但公开网站不能出现基础布局崩坏；后续是否做专门移动端交互再单独决策。

## 核心判断

公司库不应是“公司卡片墙”或统计页，而应是“投研筛选台”：

- 默认展示高密度公司表格，便于横向比较。
- 左侧提供产业链、市场、行情状态、供需状态筛选。
- 右侧展示选中公司的详情，避免用户跳转后迷路。
- 节点库、公司库、行情页、供需关系页必须读取同一份公司主数据和节点映射，不能各自维护一套公司列表。
- 最终要部署成别人可访问的网站，因此底层需要预留数据库和服务端数据更新能力。

## 数据一致性原则

所有页面共享以下核心实体：

| 实体 | 用途 |
|---|---|
| `companies` | 公司主数据：名称、代码、市场、交易所、币种、别名 |
| `subnodeCompanyCoverages` | 公司与产业链节点的映射：节点、角色、排序、说明、证据来源 |
| `marketSnapshots` | 每日行情与估值：股价、涨跌幅、市值、PE、PB、PS、成交额、交易日 |
| `supplyRelations` | 上下游关系：供应方、采购方、产品、可信度、来源 |
| `sources` | 证据来源：公告、年报、官网、研报、新闻 |
| `updateRuns` | 数据更新任务记录：来源、状态、开始时间、结束时间、错误信息 |

页面使用方式：

| 页面 | 读取数据 |
|---|---|
| 主界面 | 节点结构、节点输入输出、代表公司预览 |
| 节点库 | 节点结构、公司映射、代表公司 |
| 公司库 | 公司主数据、节点映射、行情快照、供需关系 |
| 行情数据 | 公司主数据、行情快照、历史估值 |
| 供需关系 | 公司主数据、供需关系、证据来源 |

任何公司只允许通过统一 `companyId` 关联。页面上显示的公司名称、代码、市场标签、节点归属必须来自同一份数据。

## 第一阶段：公司库静态版与统一视图模型

### 目标

在当前静态数据基础上，把公司库做成可用的投研筛选台，并让节点库里的公司展示与公司库保持一致。

### 范围

- 公司库主界面改为“筛选区 + 公司表格 + 右侧详情”。
- 表格默认按 A股优先、产业链重要性、角色优先级排序。
- 支持按公司名、股票代码、市场、产业链节点、公司角色筛选。
- 表格预留行情字段：股价、涨跌幅、市值、PE TTM、更新时间。
- 没有行情数据时显示轻量空状态，不用大段解释。
- 右侧详情展示：
  - 公司概览
  - 产业链位置
  - 相关节点
  - 行情估值占位
  - 供需关系占位
  - 证据来源占位
- 节点库里的公司列表继续从同一份公司映射读取，不允许在节点页面硬编码公司。

### 默认表格字段

| 字段 | 说明 |
|---|---|
| 公司 | 公司名、代码、市场标签 |
| 市场 | A股、港股、美股、其他 |
| 产业位置 | 对应主链和最小节点 |
| 角色 | 龙头、核心供应商、重点参与、补充观察 |
| 相关节点 | 一家公司可覆盖多个节点 |
| 股价 | 后续接行情 |
| 涨跌幅 | 后续接行情 |
| 市值 | 后续接行情 |
| PE TTM | 后续接行情 |
| 供需关系 | 客户数、供应商数、可信度状态 |

### 验收标准

- 用户进入公司库后，不看说明也能搜索和筛选公司。
- 任意节点中的公司，在公司库中能搜到。
- 任意公司详情中的产业链位置，能和节点库对应节点一致。
- 没有行情数据时，界面仍然可读，不出现大面积“待接入”文案。
- UI 不显著展示“几级节点”“覆盖数量”“映射数量”等内部结构词。

## 第二阶段：数据库与服务端数据层

### 目标

为公开网站和每日行情更新建立持久化数据层，替代纯静态种子数据。

### 数据库判断

最终要部署到网站给别人访问，因此建议使用 Postgres 兼容数据库。Supabase Postgres 是优先候选，因为它适合 Next.js 网站、公开只读查询、后续权限控制和定时任务接入。

SQLite 或 DuckDB 适合本地原型，但不适合作为多人访问网站的主数据库，除非额外维护服务端封装和部署状态。

### 范围

- 建立数据库表结构，对应：
  - `companies`
  - `company_aliases`
  - `industry_nodes`
  - `subnode_company_coverages`
  - `market_snapshots`
  - `supply_relations`
  - `sources`
  - `update_runs`
- 建立数据访问层，页面不直接依赖数据库表字段。
- 保留当前静态数据导入能力，作为初始化种子。
- 公司库、节点库、行情页、供需关系页都通过同一套查询接口读取数据。
- 为公开网站设置只读接口，避免客户端暴露写入权限和密钥。

### 验收标准

- 从数据库读取的数据能完整替代当前静态公司数据。
- 公司库、节点库显示的公司数量和节点归属一致。
- 行情数据可以按 `companyId` 和交易日查询最新值。
- 供需关系可以按公司查询上游、下游和证据来源。
- 数据库密钥不进入前端代码、不进入仓库。

## 第三阶段：每日更新与公开部署

### 目标

上线公开网站，并让行情估值数据支持每日更新。供需关系和证据数据先支持人工维护，再逐步引入自动化更新。

### 行情更新

每日更新任务需要写入：

| 字段 | 说明 |
|---|---|
| `companyId` | 公司唯一标识 |
| `tradedAt` | 交易日 |
| `price` | 收盘价或最新价 |
| `changePct` | 涨跌幅 |
| `marketCap` | 总市值 |
| `peTtm` | PE TTM |
| `pb` | PB |
| `ps` | PS |
| `turnover` | 成交额 |
| `sourceId` | 行情来源 |
| `updatedAt` | 入库时间 |

行情页和公司库只展示最新快照；后续再增加历史走势图和估值分位。

### 供需关系更新

供需关系不建议一开始完全自动化。优先维护高可信关系：

- 年报披露
- 公司公告
- 官网客户案例
- 明确新闻报道
- 高可信研报

每条关系必须有可信度状态：

- 已确认
- 高可信推断
- 待验证

### 公开部署

部署目标：

- 网站可通过公网 URL 访问。
- 默认只读访问。
- 不要求用户登录。
- 数据库和行情源密钥只在服务端环境变量中配置。
- 桌面端体验优先，移动端做到基本可访问但不专门优化。

可选部署路线：

| 路线 | 适用情况 |
|---|---|
| Vercel + Supabase | 优先候选，适合 Next.js、公开访问和托管数据库 |
| Cloudflare Pages + Supabase | 适合静态/边缘访问，服务端能力需额外确认 |
| 自有服务器 + Postgres | 控制力强，但运维成本更高 |

### 验收标准

- 外部用户可以通过公网链接访问网站。
- 公司库能展示最新一个交易日的行情数据。
- 数据更新时间在页面可见。
- 数据更新失败时，页面保留最近一次成功数据，并显示数据新鲜度。
- 密钥不暴露到浏览器、不进入仓库。
